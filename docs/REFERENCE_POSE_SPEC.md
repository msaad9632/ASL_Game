# Reference Pose System — Spec

Permanent subsystem of the Avatar Engine. A "reference pose" is a hand-posed Blender snapshot of the
avatar (same Y-Bot rig, `mixamorig:` bone names), exported as a GLB and reduced to per-bone local
rotation/translation. It is **ground truth** for one specific moment of one sign. The engine's
retargeting solver is compared against it **numerically** (angular error in degrees, positional
error in meters) — this replaces "does it look right in AvatarLab" with a real regression oracle
that runs in CI.

This is not scoped to Milestone 5. It is designed to hold reference poses for hundreds of signs
without redesign — adding a new pose never requires touching the extraction/comparison code, only
running the two CLI tools described below.

## Folder structure

```
reference_poses/                          <- repo root, source of truth, git-tracked
  blender/           *.blend               human-authored, archival only — no code reads these
  glb/                <poseId>.glb         posed GLB export from Blender — the actual ground truth
  screenshots/         <poseId>_*.png       optional human-reference images (front/side/3-4)
  metadata/            <poseId>.json        generated by extractReferencePose.ts
                        index.json          generated — list of all poseIds, for the browser viewer

web/public/reference_poses/                <- SERVABLE MIRROR, also git-tracked
  glb/                <poseId>.glb         copy of reference_poses/glb/<poseId>.glb
  metadata/            <poseId>.json        copy of reference_poses/metadata/<poseId>.json
                        index.json          copy
```

**Why two copies of `glb/` and `metadata/`?** `reference_poses/` at the repo root is the conceptual
source of truth (matches the folder layout used for planning this subsystem, keeps repo-root data
separate from the `web/` app). But the AvatarLab browser viewer can only `fetch()` files that Vite
serves as static assets, which means they must live under `web/public/`. `extractReferencePose.ts`
writes both copies in one run — you never hand-maintain the mirror.

## Reference pose metadata schema

```jsonc
{
  "poseId": "HELLO",                 // filename this is saved under
  "signName": "HELLO",               // must match a key in web/src/avatar/animation/signPaths.ts
  "frameFraction": 1.0,              // REQUIRED — 0..1, which point of the sign's authored timeline
                                      // this static pose represents. Never inferred — the person who
                                      // posed it must state it (Appendix A Rule 2: never invent
                                      // missing information).
  "sourceGlb": "glb/HELLO.glb",
  "avatarVersion": "a1b2c3d4",       // content hash of the REST-pose ybot.glb this was posed against
                                      // (for drift detection — see below)
  "generatorVersion": "extractReferencePose@1.0.0",
  "extractedAt": "2026-07-01T12:00:00.000Z",
  "bones": {
    "mixamorig:RightArm": { "rotation": [x, y, z, w], "translation": [x, y, z] },
    "mixamorig:RightForeArm": { "rotation": [x, y, z, w], "translation": [x, y, z] }
    // ... every bone the extractor found in the posed GLB
  }
}
```

`avatarVersion` mismatch is a **warning**, not a hard failure — the rig may have legitimately
changed since the pose was captured. `compareReferencePose.ts` and the Vitest suite both print/flag
this so stale references are visible, never silently compared against the wrong skeleton.

## Naming convention

- `poseId` — `UPPER_SNAKE_CASE`, matches the sign name for a single-keyframe reference (`HELLO`).
  For a future multi-keyframe sign, append a short suffix: `COFFEE_start`, `COFFEE_mid`. One
  metadata file, one GLB, one poseId — never bundle multiple poses into one file.
- Every `poseId` must map to exactly one `.glb` in `reference_poses/glb/` and one `.json` in
  `reference_poses/metadata/`.

## Comparison thresholds (current)

Defined once, in `web/src/avatar/reference/ReferencePoseCompare.ts`:

| Metric | Threshold | Why |
|---|---|---|
| Per-bone angular error | 15° | Loose today because only arm bones (`*Arm`, `*ForeArm`) are solved (Milestone 5). Fingers/palm-roll are Milestone 6+ and are reported as `unsolved`, never scored — a reference pose is never penalized for a part of the engine that doesn't exist yet. |
| Wrist / elbow positional error | 3cm | A hand-posed Blender reference and a procedurally-authored target (`signPaths.ts`) were never guaranteed to land on the exact same millimeter — this threshold exists to catch gross errors (the exact class of bug fixed in M5: a 2600cm-off elbow from a units bug), not to demand pixel-perfect agreement with hand authoring. |

Both are expected to tighten as later milestones (finger solver, animation baking, verification
engine) make the solver more complete — track threshold changes in commit messages, not silently.

## Acceptance criteria for a new reference pose to be considered "in the system"

1. `reference_poses/blender/<poseId>.blend` exists (or is otherwise archived — the source of truth
   for re-editing the pose later).
2. `reference_poses/glb/<poseId>.glb` exists and loads through `buildHierarchy()` without warnings
   about missing arm/finger chains.
3. `extractReferencePose.ts <poseId> <signName> <frameFraction>` has been run successfully (exits 0).
4. Both `reference_poses/metadata/<poseId>.json` and the `web/public/reference_poses/` mirror exist
   and are committed to git.
5. `compareReferencePose.ts <poseId>` and `referencePoseRegression.test.ts` both report the pose
   (whether PASS or FAIL — a documented, understood FAIL is acceptable; a pose the system doesn't
   know about is not).
